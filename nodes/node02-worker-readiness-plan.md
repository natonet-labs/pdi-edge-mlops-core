## Step 1 — Static IP via Netplan

> **Why this must be done on the OS, not just the router:** A DHCP reservation on the router is tied to the router — it breaks if the router is replaced, reconfigured, or the node moves. A static IP configured in netplan survives router changes, DHCP failures, and network topology changes. The OS is the source of truth for the node IP.

### 1.1 Identify the Active Ethernet Interface

The Ethernet interface name differs between machines — do not assume it.

```bash
ip link show
# Look for the enp* interface that is UP and not a virtual interface (flannel, cni0, veth*)
# panda-control: enp2s0
# panda-worker:  enp1s0
```

Then inspect the full network state:

```bash
ip addr show        # note current IP and subnet prefix
ip route show       # note default gateway
ls /etc/netplan/    # list config files
sudo cat /etc/netplan/*.yaml
```

**On panda-worker, this revealed:**
- Ethernet interface: `enp1s0`
- Current IP: `<your-static-ip>/22` (DHCP lease, `valid_lft` countdown)
- Gateway: `<your-gateway-ip>`
- Two netplan files:
  - `01-network-manager-all.yaml` — sets renderer to NetworkManager globally
  - `50-cloud-init.yaml` — controls `enp1s0` with `dhcp4: true` (cloud-init managed)
- WiFi (`wlo1`) already down — no carrier, no IP

> **Subnet is /22, not /24.** The subnet covers a wider range than a typical /24. Preserve the exact prefix from `ip addr show` when writing the static config.

### 1.2 Disable cloud-init Network Management

`50-cloud-init.yaml` is regenerated by cloud-init on reboot. Without this step, any changes to it will be overwritten.

```bash
sudo tee /etc/cloud/cloud.cfg.d/99-disable-network-config.cfg <<EOF
network: {config: disabled}
EOF
```

### 1.3 Write Static IP Config

Overwrite `50-cloud-init.yaml` with a static configuration. Keep the same IP to avoid disrupting the running K3s agent.

```bash
sudo tee /etc/netplan/50-cloud-init.yaml <<EOF
network:
  version: 2
  ethernets:
    enp1s0:
      dhcp4: false
      addresses:
        - <your-static-ip>/22
      routes:
        - to: default
          via: <your-gateway-ip>
      nameservers:
        addresses:
          - 1.1.1.1
          - 8.8.8.8
EOF
```

### 1.4 Fix Netplan File Permissions

Ubuntu 24.04 ships `01-network-manager-all.yaml` with overly open permissions. netplan warns about this and may refuse to apply in stricter future versions.

```bash
sudo chmod 600 /etc/netplan/01-network-manager-all.yaml
```

### 1.5 Apply and Verify

```bash
sudo netplan apply
# Should produce no warnings

ip addr show enp1s0
# inet line should show valid_lft forever (static, not a lease countdown)

ip route show
# default route should show proto static (not proto dhcp)

ping -c 3 panda-control.local
# Confirm connectivity to control plane is intact
```

### 1.6 Pin --node-ip in K3s Agent Service

Without `--node-ip`, K3s auto-detects the node IP from available interfaces. This works while only `enp1s0` is active, but will break if WiFi is re-enabled and picks up an IP — the same failure mode that required intervention on panda-control.

The K3s agent service uses an env file (`k3s-agent.service.env`) for `K3S_URL` and `K3S_TOKEN`, but `--node-ip` is a CLI flag — it must be added to `ExecStart` in the service file directly.

```bash
sudo nano /etc/systemd/system/k3s-agent.service
```

Change ExecStart from:
```
ExecStart=/usr/local/bin/k3s \
    agent \
```

To:
```
ExecStart=/usr/local/bin/k3s \
    agent \
    --node-ip <your-static-ip> \
```

```bash
sudo systemctl daemon-reload
sudo systemctl restart k3s-agent
sudo systemctl status k3s-agent
# Confirm active (running)
```

Verify from panda-control:
```bash
kubectl get nodes -o wide
# INTERNAL-IP for panda-worker should match the static IP
```

Expected log line in `systemctl status k3s-agent` output:
```
NodeIPs changed for the node" node="panda-worker" newNodeIPs=["<your-static-ip>"]
```

## Step 2 — Firewall Ports

UFW should already be active from OS setup. Verify and ensure all required ports are open:

```bash
sudo ufw status
```

Required ports:

| Port | Protocol | Purpose |
|---|---|---|
| 22 | TCP | SSH |
| 9100 | TCP | node-exporter — Prometheus scraping from panda-control |
| 10250 | TCP | Kubelet — K3s control plane communication |
| 8472 | UDP | Flannel VXLAN — pod-to-pod traffic |
| 51820 | UDP | Flannel WireGuard |

Open any that are missing:
```bash
sudo ufw allow 22/tcp
sudo ufw allow 9100/tcp
sudo ufw allow 10250/tcp
sudo ufw allow 8472/udp
sudo ufw allow 51820/udp
sudo ufw enable
```

> **Port 9100 is critical.** Without it, Prometheus on panda-control cannot scrape node-exporter metrics from panda-worker, and the Grafana cluster dashboard will show panda-worker metrics as `down`.

## Step 4 — Registry Config

Copy the K3s registry mirror config from panda-control so panda-worker can pull images from the local registry at `panda-control:32034`.

**From panda-control:**
```bash
scp /etc/rancher/k3s/registries.yaml <your-username>@panda-worker.local:/tmp/registries.yaml
```

**On panda-worker:**
```bash
sudo mkdir -p /etc/rancher/k3s
sudo mv /tmp/registries.yaml /etc/rancher/k3s/registries.yaml
sudo systemctl restart k3s-agent
```

**Verify the registry is reachable from panda-worker:**
```bash
curl http://panda-control:32034/v2/
# Expected: {}
```

## Step 5 — K3s Agent Join

> **Status:** Completed 2026-04-04. Documented here for reproducibility.

The K3s join token and control plane IP are required. Retrieve the token from panda-control:
```bash
# On panda-control:
sudo cat /var/lib/rancher/k3s/server/node-token
```

**Run on panda-worker:**
```bash
curl -sfL https://get.k3s.io | \
  K3S_URL=https://<panda-control-ip>:6443 \
  K3S_TOKEN=<node-token> \
  sh -
```

The agent service starts automatically as `k3s-agent.service`.

## Step 6 — Verify Cluster State

**From panda-control:**
```bash
kubectl get nodes -o wide
```

Expected:
```
NAME            STATUS   ROLES           AGE   VERSION
panda-control   Ready    control-plane   Xd    v1.34.6+k3s1
panda-worker    Ready    <none>          Xd    v1.34.6+k3s1
```

**Verify node-exporter is reachable from panda-control:**
```bash
curl http://panda-worker.local:9100/metrics | head -5
```

**Verify Prometheus is scraping panda-worker:**

Open Grafana → Explore → Prometheus → query:
```
up{job="node-exporter", instance=~".*:9100"}
```

Both instances should show value `1`.

**Verify pod scheduling works on panda-worker:**
```bash
kubectl run test-worker --image=busybox --restart=Never \
  --overrides='{"spec":{"nodeName":"panda-worker"}}' \
  -- echo "panda-worker is schedulable"
kubectl logs test-worker
kubectl delete pod test-worker
```
